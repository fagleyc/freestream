LSWT Fan Control — ABB ACS530

User documentation for the lswt package — AeroVIS device driver + PyQt6 dark-mode GUI for the two Low-Speed Wind Tunnel (North & South) fan drives.

1 · Overview & purpose

Each tunnel's fan runs on an ABB ACS530 VFD (the deployed C# calls it "ABB530"/"ACB530") over Modbus TCP, unit/slave 1. Everything protocol/calibration here was extracted from the deployed C# source Tool_LSWT_Flow_Velocity\HwControllerVelocityLSWT_ACB530.cs (TunnelVision, on the control-room PC). One app serves both tunnels via the tunnel selector; each tunnel keeps its own startup defaults.

Key improvements over the C#: one persistent, thread-safe pymodbus client (the C# reconnected per transaction — open → one register → close, every time), typed LswtError wrapping of every pymodbus error, and a smooth host-side setpoint ramp instead of the C#'s slam-to-zero "protection".

2 · Hardware & connection

Register map (wire vs FieldTalk addresses)

The C# used FieldTalk, which counts registers from 1 — every C# address is +1 vs the wire (the same lesson as this repo's other C# ports). Wire = FieldTalk − 1:

wireFieldTalk (C#)dirmeaning
01write Control: 1150 = STOP, 1151 = START (ABB drives-profile control words; FC6). C# lines 233, 237–238
12write Reference: 0–20000 scales 0–full speed; full speed = 60 Hz motor. The C# wrote the NEGATIVE of the scaled value (line 191) — a direction convention on these fans, preserved via reference_sign = −1. FC6
102103read Actual speed: current output frequency × 10 (0–600 = 0–60.0 Hz; e.g. register 452 = 45.2 Hz). C# lines 105–111, 235

Example: 30 Hz → reference counts 20000 × 30/60 = 10000 → wire value −10000 (0xD8F0) with the default sign.

Drive IPs are NOT in the C# source (runtime-configured XML). 192.168.0.1 is a placeholder — set the real North/South drive IPs on the connection bar, then Set as Defaults (per-tunnel ~/.lswt/defaults_north.json / defaults_south.json; the LSWT_DEFAULTS env var overrides the directory).

Velocity calibration (measured, ported verbatim)

calibration.FPS_AT_HZ is the C# ftPerSecVelocityToMotorHertz table (lines 59–65): a 61-point measured curve, index = motor Hz 0…60, value = tunnel velocity in ft/s — "Data obtained from LSWT experimental data" (C# comment). Endpoints: 0 Hz → 0 ft/s, 60 Hz → 105.6851 ft/s. hz_to_fps()/fps_to_hz() interpolate linearly (the table is strictly monotonic so the inverse is exact) and clamp at the ends.

Unit conversions: fps→m/s, km/h, mph use the exact physical factors (they match the C# maxima to <0.01%); Mach uses the C# ratio (0.09466 / 105.6851 — the tool's calibrated tunnel-conditions value). RPM is deliberately not displayed: the motor pole count is not in the source, so Hz is the honest motor quantity.

3 · Getting started

python run_lswt_app.py --tunnel north --sim   # fan sim, no hardware
python run_lswt_app.py --tunnel south         # real drive — ARM to act
python -m pytest tests\test_lswt.py

Sim mode (--sim / the Simulate checkbox) swaps in SimAcs530: a first-order fan model (time constant ~3 s, sim_tau_s) honoring start/stop and the reference, so the full GUI runs without hardware.

First-live-run checklist

  1. Set the real drive IP (connection bar) → Set as Defaults.
  2. Connect (read-passive) — confirm the actual-Hz register: the gauge must track the console/drive display (register = Hz × 10).
  3. Verify the reference sign at a tiny reference: arm, setpoint ~1–2 Hz, Start Fan, and confirm the fan turns the NORMAL direction. A wrong reference_sign commands REVERSE — flip it in Settings → Advanced if so.
  4. Confirm the start/stop words: Start Fan spins up, Stop Fan coasts down and the reference reads 0.
  5. Only then run real setpoints; tune ramp_hz_per_s to taste.

4 · GUI guide

Connection bar

Tunnel selector (North/South — switching loads that tunnel's saved defaults; disabled while connected) · Drive IP · Simulate · staleness readout · status lamp (DISCONNECTED / SIMULATION / LIVE (monitor) / LIVE + ARMED) · Connect / Disconnect · Set as Defaults (per-tunnel startup defaults).

Dashboard

  • Fan speed: Hz gauge (actual + setpoint against max_hz) with animated rotor.
  • Flow tiles: Actual frequency, Tunnel velocity (in the chosen unit), Setpoint, and Commanded (ramped) — the reference actually written to the drive, which ramps toward the setpoint and never step-jumps. Below: velocity-unit selector (ft/s, m/s, km/h, mph, Mach) and a ramp progress bar with a cmd → set · actual readout.
  • Strip charts: fan frequency (Actual solid, Commanded dashed) and velocity, time-linked.

Control section (bottom)

Visibly DISARMED. ARM FAN CONTROL (confirm dialog on live hardware) enables Apply Setpoint, Start Fan and Stop Fan. The Hz and velocity setpoint spins cross-update through the calibration. The ■ E-STOP is always live — arming not required.

Menus

File: Settings… (Ctrl+,), Save/Load config, Set current as defaults, Quit. Help: Documentation (this page), About.

6 · Safety & interlocks

  • ARM gating (mirrors tunnel_plc): Start Fan / Stop Fan / Apply Setpoint stay disabled until the operator explicitly ARMS fan control, with a confirm dialog on live hardware.
  • E-STOP always live — STOP word + zero reference written immediately from the calling thread, never queued; enabled regardless of arming state.
  • Host-side ramp (ramp_hz_per_s, default 2 Hz/s): the commanded reference always ramps toward the setpoint and never step-jumps the fan. This deliberately REPLACES the C#'s crude protection (any requested change >2 ft/s → command reference 0, i.e. slam the fan toward zero). fan_start() anchors the ramp at the current actual speed.
  • fan_stop() writes the STOP word and zeroes the reference.
  • Comm loss = alert, NOT auto-stop: no successful poll within stale_after_s → status STALE (red in the GUI), but the fan is deliberately not stopped — the ACS530 holds its last reference safely on its own, and auto-stopping would turn a transient network blip into an aborted run / an uncommanded flow change mid-test. The physical console and E-STOP remain the backstop.
  • Read-passive connect — connect() performs no writes, so a host reconnect never disturbs a fan already running from a previous session.

7 · Troubleshooting

symptomdiagnosis / fix
Cannot connect The default IP is a placeholder — the real North/South drive IPs were runtime-configured XML on the old control PC and are not in the source. Set the actual drive IP and save it as the tunnel's defaults.
Fan turns the wrong way at a small reference The reference_sign convention is wrong for this drive — flip it in Settings → Advanced (the C# wrote negative references; that is the shipped default).
Gauge doesn't match the drive display Confirm the actual-Hz register scaling: register = Hz × 10 (452 = 45.2 Hz). If the console shows a different quantity, verify you are reading wire register 102.
STALE in red Comm loss. The fan keeps its last reference (deliberate — see Safety). Restore the network or control from the physical console.
Setpoint appears to lag That is the host-side ramp (ramp_hz_per_s) — watch the Commanded (ramped) tile approach the setpoint. Raise the ramp rate in Settings if appropriate.
Unexpected pymodbus errors All transport errors are wrapped into LswtError — an unwrapped pymodbus timeout once killed a control thread live (repo lesson). Check the drive network path and reconnect.
