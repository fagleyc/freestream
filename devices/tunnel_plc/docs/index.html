SSWT Tunnel — Red Lion G315

User documentation for the tunnel_plc package — Freestream device driver + PyQt6 dark-mode GUI for the sub-sonic wind tunnel itself: fan speed and start/stop.

1 · Overview & purpose

The tunnel is run by a Red Lion G315 HMI at 192.168.1.50 acting as a Modbus TCP slave (port 502, unit 1). The G315 bridges a GE VersaMax PLC (fans, heater, status lights; Ethernet SRTP) and a GE FanDrive (RPM setpoint/actual; RS-485 SNP), re-exporting the PLC tags as Crimson 32-bit "L4" gateway blocks.

Freestream ── Modbus TCP :502 ──> Red Lion G315 (.50)
                                    ├─ GE SRTP :18245 ──> VersaMax PLC (.31)
                                    │    fans, heater, status lights
                                    └─ RS-485 SNP ──> GE FanDrive
                                         RPM_Set (R00102), Actual_RPM (R00200)
We talk ONLY to the G315. The FanDrive has no network path (SNP is single-master — never attach a second serial master), and going SRTP-direct to the VersaMax would bypass the HMI's existing logic.

The architecture is two strictly separated classes: TunnelMonitor (read-only poller — has no write methods, a test enforces this) and TunnelControl (the only write path — refuses to instantiate without enable_writes=True; in the GUI it does not even exist until the operator ARMS writes).

2 · Hardware & connection

Register map (authoritative: read_mappings.txt / write_mappings.txt)

Crimson L4 gateway blocks: each element = two 16-bit holding registers; element N lives at protocol address 2·(N−1).

Block1 — READ, elements 1–16 @ address 0 (32 registers, one atomic FC3 read)

No analog channels are in the gateway — pressures/temps stay with the DAQ devices; Mach/Reynolds are computed by Freestream from its own DAQ channels.

Verified live 2026-07-07 (probe_tunnel.py, read-only)

  • Word order is low_first (RPM_Set=600 and three lit booleans all put the value in the low word); Actual_RPM read 0 at idle, all booleans clean 0/1.
  • RPM registers are Crimson fixed-point ×10 (1 display decimal): register 600 = HMI display 60.0 RPM → rpm_scale = 0.1. rpm_max and all driver APIs are in true engineering RPM.

Bearing temperatures (opt-in Block1 extension — NOT yet in Crimson)

The analog bearing sensors are VersaMax AI0007–AI0009 (Analog_Feedback.B1/B2/B3); today only the Bearing_Temp_Low_Light boolean reaches the gateway. To enable the analogs: ① in Crimson 3, extend the read gateway block with elements 17/18/19 mapped to Analog_Feedback.B1/B2/B3 (direction 'Tag to Block', landing at protocol addresses 32/34/36) and re-download to the G315; ② enable Bearing temperatures in File → Settings. The Block1 poll then becomes ONE contiguous 38-register FC3 read.

Unit not confirmed. Scaling is linear per channel from tunnel_tags.csv (B1 raw 955–5035 → 0–150, B2 969–4979 → 0–150, B3 930–4994 → 0–150). The unit label defaults to °C but the cal vintage/unit has not been confirmed on the rig — it could equally be °F. Verify against the bearing RTD spec before trusting absolute values; the label is editable in Settings.

3 · Getting started

python run_tunnel_plc_app.py --sim   # plant sim, no hardware
python run_tunnel_plc_app.py         # real gateway — read-only until ARMED
python -m tunnel_plc.probe_tunnel    # READ-ONLY probe / word-order check
python -m pytest tests\test_tunnel.py

Simulate mode swaps in SimGateway: fan spool-up, fault injection, comm-failure injection and a write history — the full GUI runs with no hardware.

Manual write-validation procedure (NOT yet performed)

The write path is sim-tested only. Before first operational use, with a person at the console and the test section clear:

  1. RPM setpoint, fan OFF: set rpm_max (Settings), arm, command a small RPM (e.g. 100). Confirm the HMI's setpoint display follows and RPM_Set in Block1 reads back the same value — validates Block2 element 105 end-to-end without motion.
  2. Momentary pulse: command Cooling fan START (the benign load) and watch the VersaMax/HMI: the button must latch the fan on from a single 250 ms pulse, exactly like a touchscreen press. If it needs a longer hold, raise button_hold_ms; when satisfied, set momentary_verified in Settings to silence the per-pulse warnings.
  3. Tunnel fan start/stop: only after 1–2 pass, low RPM first, and verify stop works before anything else matters.
  4. Record the verified rpm_max in a saved config JSON.

4 · GUI guide

Connection bar

Gateway IP · Simulate · data-age/staleness readout (data 0.4s turns to a red STALE Ns) · status lamp (DISCONNECTED / SIMULATION / LIVE (read-only) / LIVE + ARMED) · Connect / Disconnect.

Dashboard

  • Fan speed: RPM gauge (actual + setpoint marker against rpm_max) with an animated fan rotor.
  • Status lights (VersaMax): Fan running, Console control, INVERTER FAULT, OIL LEVEL LOW, Bearing temp low, Bearing heater, and the four fan start/stop lights — fault lights render in the danger color when lit. Below them the three bearing-temperature readouts ("—" until the opt-in extension is enabled).
  • RPM history: Actual RPM (solid) and RPM set (dashed) vs time.

Control section (bottom)

Visibly DISARMED by default. ARM WRITES requires a connection, a configured rpm_max (> 0) and a Yes on the confirm dialog; only then is a TunnelControl constructed. Armed controls: RPM setpoint + Apply RPM (spin box clamped to rpm_max), Tunnel fan START/STOP, Cooling fan START/STOP. Every write lands in the write-log table (Time, Tag, Old → New, Note). Disconnecting, a comm watchdog drop, or unchecking ARM destroys the control object again.

Menus

File: Settings… (Ctrl+,), Save/Load config, Quit. Help: Documentation (this page), About.

6 · Safety & interlocks

  • Monitor/control separation — TunnelMonitor has no write methods (test-enforced); TunnelControl refuses to instantiate without keyword-only enable_writes=True, and in the GUI no write-capable object exists until the operator ARMS.
  • ARM gating — arming requires a live connection, a configured rpm_max and an explicit confirmation dialog naming the real machinery.
  • RPM clamp — setpoints clamp to rpm_max; all RPM writes are refused while rpm_max is 0 (the shipped default).
  • Fault/staleness interlock — commands are refused while the snapshot is stale or Inverter_Fault_Light is set. Deliberate exception: the two stop buttons bypass those interlocks (stopping is the safe direction) and only require a live gateway.
  • Momentary buttons — fan buttons are momentary: write 1, hold 250 ms, write 0 — mirroring a touchscreen press.
  • Write audit — every write is logged (timestamp, old, new) to write_log and the GUI table.

7 · Troubleshooting

symptomdiagnosis / fix
Every control press raises ExceptionResponse(function_code=144, exception_code=2) (seen live 2026-07-07) FC16 write rejected, ILLEGAL DATA ADDRESS — the slave serves the registers for reading but refuses writes there. Fix on the Crimson side (open SSWT_Logger_G315_v2 in Crimson 3): ① Communications → Modbus TCP slave driver: 'Read Only' must be NO; ② the Block2 gateway block (elements 101–105) must have direction 'Block to Tag' (Block1 stays 'Tag to Block'); ③ re-download to the G315. Then run python tunnel_write_check.py --arm — it writes the SAME retained RPM_Set value back (a plant no-op) and reports FC16/FC6 acceptance + readback. The driver itself tries FC16 and falls back to FC6 automatically, and write rejections surface in the status bar with this diagnosis.
STALE data / commands refused No successful Block1 poll within stale_after_s. The monitor reconnects with exponential backoff — check the network path to 192.168.1.50. Stop buttons still work while connected.
Cannot ARM rpm_max is 0 (not configured) — set the real fan speed limit in File → Settings first; arming also requires a running monitor.
Bearing temperatures show "—" The opt-in extension is off, or the Crimson block was never extended with elements 17–19 (see §2). Both are required.
Fan button pulse doesn't latch the fan The 250 ms pulse may be too short for the VersaMax logic — raise button_hold_ms and repeat the supervised momentary test (§3, step 2).
