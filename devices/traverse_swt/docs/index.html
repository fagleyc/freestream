SSWT Traverse — WAGO 750

User documentation for the traverse_swt package — standalone AeroVIS device driver + PyQt6 dark-mode GUI for the sub-sonic wind tunnel (SSWT) 3-axis probe traverse.

1 · Overview & purpose

The SSWT traverse is driven by a WAGO 750 controller at 192.168.1.21:502 (Modbus TCP, unit 1) with three 750-673 stepper modules — X axial, Y lateral, Z vertical. This app replaces the deployed C# tool (Tool_SSWT_Traverse, FieldTalk Modbus) and was reverse-engineered 2026-07-07 from the CoDeSys 2.3 PLC source (LVW_V3_2021.pro) and the deployed C#.

Improvements over the C#: one persistent Modbus connection (the C# reconnected before every operation), one 16-register block read per tick, change-only ControlWord writes, host-side calibrated positioning, a clean 1,000,000-count rollover unwrap into absolute position, host-side limit-switch reaction and host-side homing.

Motion model. The PLC runs the steppers in velocity mode at a fixed ±2000 steps/s (accel/decel baked into the PLC program). The host has direction bits only — no speed, no position mode. Positioning is a host-side bang-bang loop: command a direction, watch the counts, drop the bit inside the tolerance band (default 0.02").

2 · Hardware & connection

Register map (wire = 0-based, what pymodbus uses)

The C# FieldTalk library counts registers from 1, so every C# address is +1 vs the wire.

wire%MWC# addrmeaning
12288MW012289 ControlWord — host writes (FC6)
12289MW112290* StatusWord — limit bits 0/1/2 = X/Y/Z
12298MW1012299 X position, DINT low-word-first (FC3/FC4)
12300MW1212301Y position, DINT
12302MW1412303Z position, DINT

* The C# declared 12290 as a "home button" register but never used it — and it lands on the StatusWord anyway. There is no homing in the PLC (Homing := FALSE everywhere); homing is host-side (see Safety).

ControlWord bits (PLC source: ControlWord AT %MW0)

bitPLC namebitPLC name bitPLC name
0AxialJogFwd2LateralJogFwd 4VerticalJogFwd
1AxialJogRev3LateralJogRev 5VerticalJogRev

The "Jog" names are the PLC's — the host uses these direction bits for bang-bang move_to, not a software jog. Bits 6/7/8 were the per-axis stepper BasicReset; the reset pulse was removed as non-functional on the rig.

Swapped labels. The C# labeled Y and Z with fwd/rev swapped vs the PLC names (C# "Y forward" = bit3 = PLC LateralJogRev; C# "Z forward" = bit5 = PLC VerticalJogRev). The default masks reproduce the C# behavior — what operators knew; each axis has a fwd_increases_counts flag to flip the sense, and move_to has a wrong-way trip that stops the axis if the target error grows.

StatusWord limit bits

The limit switches were fixed (inverted at the module) in 2026-07 and are live again. The PLC copies them into StatusWord %MW1: bit0 = X/Axial, bit1 = Y/Lateral, bit2 = Z/Vertical — each the axis's negative-direction switch. Polarity (rig-verified 2026-07-22): the bits are ACTIVE-LOW — the NC chain drives each bit HIGH in the healthy state; the bit clears when the switch is engaged (limit_active_low = True, the default). X/Axial's limit input is DISABLED (x.limit_enabled = False, per the rig): its bit is ignored entirely — no runtime reaction and no homing on X.

Position counter — 1M rollover + absolute tracking

The 750-673 modules are configured to roll their position counter over cleanly at 1,000,000 counts (unsigned 0…999,999) on all three axes, so the module never approaches the 24-bit counter limit where it used to silently stop stepping (observed live 2026-07-07). The driver unwraps the 1M ring into a continuous absolute position (shortest-path modular delta each 50 ms tick), so a single move can cross any number of wrap boundaries. A counter-jump guard (max_counts_per_tick, default 100,000) treats any larger raw delta as a counter event (module reset/power event): the position is held and the raw baseline re-based instead of integrating a phantom move.

DINT word order: CoDeSys stores the low word at the lower %MW; the C# FieldTalk default was also low-word-first, and probe_traverse.py verifies it live.

Calibration

C# two-point form, per axis, signed slope: inches = inch_high − (counts_high − counts) / clicks_per_inch. Default slopes derive from the rig's sswtTraverseCalibrationFile.xml:

axisclicks/inch (default)soft limits (default) homing
X — Axial+13 705.6−5" … +5" none (limit input disabled)
Y — Lateral−14 841.0−18" … +18" datum −18" at the negative switch
Z — Vertical+986 938.4−25" … +25" datum +18" at the switch (sign convention set on the rig 2026-07-22)

The 750-673 position counter zeroes at PLC power-up, so the offset is per-power-cycle: axes start calibrated=False. Homing Y/Z sets their offsets; any axis can also be re-zeroed against a known position (Calibration tab, routine 2). Import legacy C# cal XML… pulls signed slopes from the old file.

3 · Getting started

python run_traverse_swt_app.py --sim   # plant sim, no hardware
python run_traverse_swt_app.py         # real PLC (192.168.1.21)
python -m traverse_swt.probe_traverse  # READ-ONLY live probe
python -m pytest tests\test_traverse.py tests\test_traverse_homing.py

Simulation mode (--sim or the Simulate checkbox) swaps in SimPlc: a velocity-integrator plant model on the unsigned 1M ring at the fixed ~2000 counts/s PLC speed, with per-axis negative-end limit switches and an injectable stall fault — homing runs end-to-end in sim.

First live checklist (supervised)

  1. probe_traverse.py — verify the register map, word order, and FC3 = FC4 before any motion.
  2. GUI, live, one axis: move the stage a known amount from the physical console and confirm the counts track it (and the direction). Counts falling as inches rise is just the signed slope — captured by calibration.
  3. Re-zero the calibration, then a short move_to — the wrong-way trip guards the first one.
  4. Homing: verify the seek direction live on the first supervised run — the seek jogs toward the negative limit at fixed PLC speed.
Per-power-cycle: the module counter zeroes at power-up, so the homing/calibration offset dies with it — re-home each setup. The offset persists only if saved ("Set as Defaults" / Save config).
TODO (rig): the real travel of Y and Z from the −18" datum is unknown — measure the real ranges from the datum on the rig and set them in Settings.

4 · GUI guide

Connection bar

WAGO PLC IP · Simulate checkbox · status lamp (DISCONNECTED / SIMULATION / LIVE) · Connect / Disconnect · Set as Defaults (saves the CURRENT settings, live calibration included, as the startup defaults, auto-loaded at every launch — separate from File → Save/Load config files).

Motion tab

  • Axis cards (X | Y | Z): big readout (raw counts while uncalibrated — move the stage from the physical console; inches once calibrated), state label (idle / → target / HOME <phase> / LIMIT / fault), target spin + Move (unlocked when calibrated), Home (Y/Z only), and an emphatic per-axis ■ STOP. The card footer shows the homing state and the raw 750-673 module status bytes S1·S2·S3.
  • All axes column: ■ E-STOP (dominant) and Stop all — both stop all three axes; either aborts a homing cycle at any point.
  • Position history plot: all enabled axes vs time; plots raw counts until every enabled axis is calibrated, then inches.

Diagnostics tab

Raw ControlWord echo (confirm the PLC receives the commanded bits), per-axis 750-673 module status bytes, and the module event log — every S1 transition is logged with time, position and direction. Bit meanings are undocumented in the extracted source; one faulting start on the rig identifies the error bit empirically.

Calibration tab

Routine 1: two-point calibration (slope + offset). Routine 2: offset-only re-zero against a known position. Import legacy C# cal XML… re-derives signed slopes from the old sswtTraverseCalibrationFile.xml (offsets are NOT imported — they are stale by design; re-home/re-zero on the rig instead).

Menus

File: Settings… (Ctrl+,), Save/Load config, Set current as defaults, Quit. Help: Documentation (this page), About.

6 · Safety & interlocks

The host is the only protection. The module hardware-limit lockout has been UNLINKED (Ptr_LimitSwitch = 0): the 750-673 never hard-stops or latches on a limit by itself. The driver reads the StatusWord every control tick and reacts host-side.
  • E-STOP / Stop all — stops all axes immediately, issued from the calling thread; aborts any homing cycle and leaves the axes stopped. Per-axis ■ STOP does the same for one axis. Stops are always forced through (never deferred by the change-only write logic, never delayed by the direction dwell).
  • Runtime limit trip — if an axis's switch engages (bit clears) while the axis is commanded TOWARD the negative limit (outside homing), the host stops the axis within one control tick and flags a LIMIT fault. Commanding AWAY from a made switch is allowed — that is the recovery path — and the fault clears with the switch.
  • Homing (Y/Z) — host-side sequence: SEEK toward the negative limit polling the StatusWord bit every ~15 ms (reaction bound ≈2000 steps/s × 15 ms ≈ 30 counts of overtravel) → stop → BACKOFF the opposite way until the bit clears + home_backoff_margin_s → settle → calibrate_offset(home_datum_in). Both phases have timeouts that fault cleanly (axis stopped, homed stays False).
  • Soft limits — calibrated min_in/max_in bound every move_to.
  • Wrong-way trip — a move whose error grows for wrongway_ticks consecutive ticks aborts (guards a wrong direction sense).
  • Stall abort — a commanded axis with frozen counts warns at ~1 s and is aborted at ~3 s (a faulted module must not stay commanded — it would lurch when the fault clears).
  • Direction-change dwell — any start or reversal passes through a commanded stop ≥600 ms so a module is never hit with a conflicting command mid-sequence (the likely start/stop fault cause).
  • Oscillation guard — more than max_reversals re-reversals around the target aborts the move instead of ping-ponging the motor.
  • Comm watchdog — max_consecutive_errors failures → stop all + disconnect. All transport exceptions are wrapped; the control loop has a catch-all that emergency-stops + disconnects instead of dying (hardening from a live crash).

7 · Troubleshooting

symptomdiagnosis / fix
Y (lateral) does not move; STALL warning then abort Diagnosis order: ① Diagnostics tab → ControlWord echo: bits 2/3 must appear while a Y move runs (confirms the PLC received the command). ② Frozen counts = the 750-673 module is not issuing steps (the counter is the module's own step count) → module faulted/disabled; check the module's error LEDs in the cabinet. ③ Counts moving but no physical motion = motor power/wiring (open-loop stepper — the module can count steps into a dead motor circuit). ④ If it does not clear, the lateral drive may need a PLC power cycle (the CoDeSys program has MC3_RestoreDefault plumbing for the lateral module, suggesting past trouble).
Move aborts immediately with a wrong-way message The axis's fwd_increases_counts sense is wrong — flip it in Settings (this is exactly how Z was found: move_to(z=+1") wrong-way-tripped twice; the fix is now the default).
LIMIT fault on an axis The negative-direction switch is engaged. Command AWAY from the switch to recover; the fault clears with the switch. If limits behave inverted, check limit_active_low (Settings ▸ Advanced) — but the rig-verified default is active-low.
Position jumps / resets to zero Module power event — the 750-673 counter zeroes at power-up. The counter-jump guard re-bases instead of integrating the jump; the axis drops to uncalibrated. Re-home (Y/Z) or re-zero (X).
Modbus timeouts / gateway errors The watchdog stops all axes and disconnects after 5 consecutive errors. Check the network path to 192.168.1.21; reconnect from the GUI.
Module status bytes S1 (Diagnostics) Observed live: 0x00 idle, 0x03 starting, 0x1B running, 0x01 stopping, 0x1F transient at start. No distinct code at the old counter-limit stop.
Software jog / stepper-reset buttons missing Removed 2026-07 as non-functional on the rig; the retired PLC-side homing POUs live in plc_homing_ARCHIVED/ (never downloaded).
